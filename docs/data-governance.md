"""Data Governance and Lineage Tracking

## Overview

This document outlines data governance practices for the Code Explainer project,
including data provenance, lineage tracking, retention policies, and compliance.

## Data Classification

### Training Data
- **Source**: Public repositories (GitHub, GitLab, Hugging Face Datasets)
- **Content**: Code snippets and natural language explanations
- **Sensitivity**: Low (public domain)
- **Retention**: Permanent (archived for reproducibility)
- **License**: CC-BY-4.0 or compatible

### User Code (Runtime)
- **Source**: API requests or batch uploads
- **Content**: User-provided source code for explanation
- **Sensitivity**: High (proprietary)
- **Retention**: 30 days (configurable via CODE_EXPLAINER_DATA_RETENTION_DAYS)
- **License**: User retains ownership

### Model Outputs (Explanations)
- **Source**: Generated by CodeExplainer models
- **Content**: Natural language explanations
- **Sensitivity**: Medium (derived from models)
- **Retention**: Same as input code (30 days default)
- **License**: User retains ownership

### Evaluation Data
- **Source**: Benchmark datasets (OPEN_EVALS_SPEC.md)
- **Content**: Curated code and human-judged explanations
- **Sensitivity**: Low (public benchmarks)
- **Retention**: Permanent
- **License**: As specified in OPEN_EVALS_SPEC.md

## Data Provenance

All data lineage is tracked via INTAKE_TEMPLATE.yaml:

```yaml
# data/INTAKE_TEMPLATE.yaml
dataset:
  name: string          # Dataset identifier
  source: string        # Origin URL or path
  version: string       # Dataset version
  created_at: string    # ISO-8601 timestamp
  
provenance:
  contributors: [string]  # Data creators/curators
  license: string         # License identifier (SPDX)
  attribution: string     # Required attribution text
  
quality:
  records_total: int
  languages: [string]
  code_samples: int
  explanation_coverage: float  # 0.0-1.0
  
lineage:
  upstream: [string]    # Sources this dataset depends on
  downstream: [string]  # Datasets derived from this one
  transformations: [string]  # Data processing steps
```

## Data Retention Policies

### Automatic Cleanup

Scheduled cleanup job removes data older than retention threshold:

```bash
# Run daily cleanup (e.g., via cron or scheduler)
python scripts/cleanup_old_data.py --retention-days 30
```

Environment variables:
- `CODE_EXPLAINER_DATA_RETENTION_DAYS`: Days to keep user data (default: 30)
- `CODE_EXPLAINER_CLEANUP_ENABLED`: Enable automatic cleanup (default: true)
- `CODE_EXPLAINER_CLEANUP_SCHEDULE`: Cron expression for cleanup (default: "0 2 * * *")

### Manual Data Deletion

For individual requests, delete on-demand:

```python
from code_explainer.data_governance import delete_user_request

# Delete all data for request ID
delete_user_request("request-id-12345")

# Delete by time range
delete_by_date_range(start_date="2024-01-01", end_date="2024-02-01")
```

## Compliance

### GDPR Compliance
- Right to erasure: Users can request complete data deletion
- Data minimization: Only code and explanations stored, no metadata
- Transparency: Data retention policy documented and enforced

### Privacy Controls
- Opt-out mode: Set `CODE_EXPLAINER_DATA_STORAGE_DISABLED=1` to disable persistence
- Encryption at rest: Sensitive data encrypted with user-provided keys (optional)
- Encryption in transit: TLS 1.3 for all API requests

### Audit Logging
All data access is logged to `logs/data-audit.log`:

```
2024-01-15T10:30:45Z | REQUEST-12345 | STORE | code_snippet | 1024 bytes
2024-01-15T10:30:46Z | REQUEST-12345 | RETRIEVE | explanation | 256 bytes
2024-01-20T02:15:00Z | CLEANUP-BATCH | DELETE | 152 items | retention_expired
```

Each dataset must include a provenance card in `data/provenance/`:

```markdown

## Description
1,000 curated Python code snippets with expert explanations

## Source
- GitHub repositories filtered by: MIT license, 100+ stars, active maintenance
- Time period: 2020-2024

## Composition
- Total samples: 1,000
- Languages: Python (100%)
- Code length: mean=45 lines, median=20 lines, max=500 lines
- Explanation length: mean=150 words, median=120 words

## Quality
- Duplicate detection: < 1% duplicates (exact + fuzzy match)
- Expert review: 100% human-reviewed by 3+ judges
- Inter-rater agreement: Cohen's κ = 0.87

## License & Attribution
- License: CC-BY-4.0
- Citation: @dataset{python_explanations_2024, ...}

## Limitations & Biases
- Bias toward popular Python frameworks (Django, NumPy)
- Limited edge-case handling code
- May not represent newer language features (3.12+)

## Ethical Considerations
- All code filtered for malicious patterns
- No personally identifiable information (PII)
- No security vulnerabilities intentionally included

## Related Datasets
- Upstream: GitHub API dataset (2024-01-15)
- Downstream: python-explanations-eval-1 (for model evaluation)
```

## Data Governance Roles

### Data Steward
- Maintains provenance records
- Enforces retention policies
- Approves new datasets

### Data Engineer
- Implements cleanup scripts
- Monitors data quality
- Handles GDPR requests

### Model Owner
- Evaluates training data quality
- Documents model-specific biases
- Approves model releases

## Tools & Integration

### Data Version Control
Use DVC (Data Version Control) for large datasets:

```bash
dvc add data/train.json
dvc push
```

### Data Lineage Tracking
Automatic lineage tracking via `src/code_explainer/data_governance.py`:

```python
from code_explainer.data_governance import log_data_lineage

log_data_lineage(
    operation="model_training",
    input_datasets=["train.json"],
    output_datasets=["model-v1.safetensors"],
    timestamp="2024-01-15T10:30:00Z"
)
```
